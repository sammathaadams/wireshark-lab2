================================================================================
STANDARD OPERATING PROCEDURE: WIRESHARK NETWORK TRAFFIC ANALYSIS LAB
================================================================================

This Standard Operating Procedure (SOP) provides an end-to-end guide for
provisioning a Windows Server 2025 Azure VM via Azure CLI, installing Wireshark,
capturing live network traffic, applying display filters, and performing guided
protocol analysis exercises — including DNS inspection, TCP handshake observation,
cleartext credential capture, and full TCP stream reconstruction.

Certification Alignment: CompTIA Network+ | Security+ | CySA+
Career Relevance:        Network Engineer | SOC Analyst | Cloud Security Engineer
Platform:                Microsoft Azure — Windows Server 2025 VM via Azure CLI
Estimated Cost:          ~$0.05–$0.10/hr for Standard_D2s_v3 while running
Time to Complete:        2–4 hours across multiple sessions

--------------------------------------------------------------------------------
SECTION 1: LOCAL REPOSITORY SETUP (VS CODE TERMINAL)
--------------------------------------------------------------------------------
Before provisioning the Azure environment, initialize a local Git repository on
your workstation to track captures, scripts, and documentation.

1. Open VS Code and launch a new terminal (Ctrl + ~).

2. Create and navigate to your project directory:

   mkdir wireshark-lab2
   cd wireshark-lab2

   * mkdir: Creates a new folder.
   * cd: Changes the terminal's working directory into that folder.

3. Initialize the Git repository:

   git init

   * git init: Creates a hidden .git directory, turning the folder into a
     version-controlled repository.

4. Scaffold the project structure:

   echo "# Wireshark Network Analysis Lab" > README.md
   mkdir scripts screenshots

5. Stage and commit the baseline:

   git add .
   git commit -m "initial commit: wireshark lab project structure"

   * git add .: Stages all new files for the next commit snapshot.
   * git commit: Permanently records the staged state into local Git history.

SECTION 2: DEPLOY THE AZURE VM (AZURE CLI)
--------------------------------------------------------------------------------
Run the following from your local terminal (Azure CLI must be installed and
authenticated via "az login" before proceeding).

PHASE 1: CREATE THE RESOURCE GROUP
------------------------------------
az group create --name rg-lab02-0626 --location westus2

* az group create: Provisions a new Azure Resource Group — a logical container
  for all resources in this lab.
* --name rg-lab02-0626: Resource group name. The date suffix makes it easy to
  identify and clean up after the lab.
* --location westus2: Azure region — West US 2.

PHASE 2: DEPLOY THE WINDOWS SERVER 2025 VM
-------------------------------------------
az vm create `
  --resource-group rg-lab02-0626 `
  --name ws01 `
  --image MicrosoftWindowsServer:WindowsServer:2025-datacenter-g2:latest `
  --admin-username azureuser `
  --admin-password 'YourStrongPassword123!' `
  --public-ip-sku Standard `
  --size Standard_D2s_v3

NOTE: The backtick (`) is PowerShell's line continuation character.
If running from Bash/WSL, replace each backtick with a backslash (\).

* az vm create: Provisions the VM along with a VNet (ws01-vnet) and NSG
  (ws01-nsg) automatically.
* --image MicrosoftWindowsServer:WindowsServer:2025-datacenter-g2:latest: Deploys
  Windows Server 2025 Datacenter using the full marketplace URN (Publisher:Offer:SKU:Version).
  This format is required when a short alias is not available for the image.
* --admin-username / --admin-password: Local administrator credentials. Use a
  strong password — this VM will have RDP open to the internet.
* --public-ip-sku Standard: Allocates a static Standard-tier public IP.
* --size Standard_D2s_v3: 2 vCPUs, 8 GiB RAM — sufficient for Wireshark GUI.

PHASE 3: OPEN RDP PORT AND RETRIEVE PUBLIC IP
----------------------------------------------
# Open port 3389 for Remote Desktop access
az vm open-port --resource-group rg-lab02-0626 --name ws01 --port 3389

NOTE: When "az vm create" completes it prints a JSON block to the terminal.
Copy the value of "publicIpAddress" from that output — you will need it for RDP.

* az vm open-port: Adds an inbound NSG rule allowing TCP 3389 from any source.

NOTE: After the deployment completes, allow 1–2 minutes for Windows to fully
boot before attempting an RDP connection.

SECTION 3: CONNECT VIA REMOTE DESKTOP (RDP)
--------------------------------------------------------------------------------
1. Confirm the VM is in a running state:
   az vm show --resource-group rg-lab02-0626 --name ws01 --query "powerState" -d -o tsv
   Expected output: VM running

2. Open Remote Desktop Connection on your local machine:
   Windows: Start → type "mstsc" → press Enter
   macOS:   Download "Microsoft Remote Desktop" from the App Store

3. Enter the public IP address retrieved in Section 2 Phase 3

4. Authenticate with:
   Username: azureuser
   Password: the password you set in Phase 2

5. Accept the certificate warning — this is expected for lab environments

SECTION 4: INSTALL WIRESHARK ON THE VM (INSIDE RDP SESSION)
--------------------------------------------------------------------------------
Once connected to ws01, open POWERSHELL AS AN ADMINISTRATOR and install
Wireshark using the official installer (Npcap is bundled — no separate download needed):

# Download the Wireshark installer — Npcap is bundled and installs in the same wizard
Invoke-WebRequest -Uri "https://2.na.dl.wireshark.org/win64/Wireshark-latest-x64.exe" -OutFile "$env:TEMP\wireshark-installer.exe"
Start-Process -FilePath "$env:TEMP\wireshark-installer.exe" -Wait

# Verify the installation
& "C:\Program Files\Wireshark\tshark.exe" --version

* Invoke-WebRequest: Downloads the latest Wireshark x64 installer from the
  official Wireshark mirror.
* Start-Process -Wait: Launches the installer GUI and holds the terminal until
  the wizard completes.
* Accept all defaults in the installer wizard. Npcap is a pre-checked component
  in the wizard and installs automatically — no separate download required.
* tshark --version: Confirms both Wireshark and the tshark CLI are installed.

SECTION 5: THE WIRESHARK INTERFACE (ORIENTATION)
--------------------------------------------------------------------------------
When you open Wireshark you will see five main areas:

  Welcome Screen      — Lists all available network interfaces. Each interface
                        shows a live wave graph indicating current traffic
                        volume. Double-click any interface to begin capturing.

  Packet List Pane    — The top panel during a capture. Each row is one packet.
                        Columns show: packet number, timestamp, source IP,
                        destination IP, protocol, length, and a summary in
                        the Info column.

  Packet Detail Pane  — The middle panel. When you click a packet in the list,
                        this panel shows the full decoded protocol stack —
                        Ethernet frame, IP header, TCP/UDP header, and
                        application layer payload. Click any arrow to expand
                        a layer and inspect individual fields.

  Packet Bytes Pane   — The bottom panel. Shows the raw hex and ASCII
                        representation of the selected packet. Most exercises
                        in this lab do not require this pane.

  Filter Bar          — The input bar at the top of the packet list. Type a
                        display filter here and press Enter to narrow the
                        packet list to only matching traffic.

KEY DISTINCTION: Display Filters vs Capture Filters
  Capture Filters:  Applied BEFORE the capture begins. They limit what gets
                    recorded to disk. Use only when you know exactly what you
                    want and need to reduce file size.
  Display Filters:  Applied AFTER the capture. They limit what is VISIBLE
                    without discarding captured data. For this lab, always use
                    display filters — they allow you to re-examine the same
                    capture through multiple lenses without re-capturing.

--------------------------------------------------------------------------------
SECTION 6: ESSENTIAL DISPLAY FILTERS REFERENCE
--------------------------------------------------------------------------------
Type any of these into the Wireshark filter bar and press Enter.

  Filter                      What It Shows
  --------------------------  -----------------------------------------------
  dns                         All DNS queries and responses
  http                        Unencrypted HTTP traffic only
  tcp                         All TCP traffic
  tcp.flags.syn == 1          TCP SYN packets (connection attempts only)
  tcp.flags.reset == 1        TCP RST packets (refused/forcibly closed connections)
  icmp                        All ICMP traffic including ping
  ip.addr == 192.168.1.1      All traffic to or from a specific IP address
  ip.src == 10.0.0.5          Outbound traffic from one specific source IP
  tcp.port == 443             All traffic on port 443 (HTTPS)
  http.request                HTTP GET and POST requests only
  http.request.method == POST HTTP POST requests only (login form submissions)

  COMBINING FILTERS
  -----------------
  Use "and" to require multiple conditions:
    tcp and ip.addr == 93.184.216.34
    This shows only TCP traffic to or from that specific IP.

  Use "or" to match either condition:
    dns or icmp
    This shows DNS and ICMP packets together.

  Use "!" to exclude:
    !arp
    This hides ARP packets, which are common background noise in captures.

SECTION 7: EXERCISE A — CAPTURE A DNS LOOKUP
--------------------------------------------------------------------------------
PURPOSE: Observe a DNS query and its response at the packet level. Understand
the structure of an A record lookup and confirm the DNS transaction visually.

BACKGROUND
  DNS (Domain Name System) translates human-readable domain names into IP
  addresses. Every website visit, API call, and email delivery begins with a
  DNS query. In Wireshark, you can see this query leave your machine and the
  response come back — with the IP address inside.

  nslookup is a built-in command-line tool that performs a manual DNS query.
  It is used here to trigger DNS traffic on demand so Wireshark can capture it.
  Run nslookup from your terminal (Command Prompt on Windows, Terminal on Mac/
  Linux) — NOT inside Wireshark. Wireshark is a capture and analysis tool only;
  it has no terminal.

STEPS
  1.  Open Wireshark
  2.  On the welcome screen, double-click your active network interface
      (the one with the most wave graph activity — usually "Ethernet" or "Wi-Fi")
  3.  Wireshark begins capturing immediately — packets start appearing in real time
  4.  Open a separate terminal window (do NOT stop the Wireshark capture)
         Windows: Windows key → type cmd → press Enter
         Mac:     Cmd + Space → type Terminal → press Enter
         Linux:   Ctrl + Alt + T
  5.  In the terminal, run:
         nslookup google.com
      The terminal will display the IP address(es) DNS returned for google.com
  6.  Switch back to Wireshark and click the red square Stop button in the toolbar
  7.  In the filter bar, type:
         dns
      Press Enter. The packet list now shows only DNS traffic.
  8.  In the Info column, find:
         "Standard query A google.com"       ← this is your outbound query
         "Standard query response A google.com" ← this is the DNS server's reply
  9.  Click the response packet to select it
  10. In the Packet Detail pane, expand "Domain Name System (response)"
  11. Expand the "Answers" section — you will see the A record entry
  12. Confirm the IP address shown matches what your terminal displayed

WHAT TO RECORD FOR YOUR PORTFOLIO
  - Screenshot of the DNS query packet with the Info column visible
  - Screenshot of the DNS response packet with the Answers section expanded
  - Note the matching transaction IDs in both packets

WHAT THIS DEMONSTRATES IN PRODUCTION
  Unexpected DNS queries to unknown or unusual domains in a capture are often
  the first indicator of malware phoning home to a command-and-control server.
  SOC analysts apply the dns filter as a first step in any suspicious-host
  investigation.

SECTION 8: EXERCISE B — WATCH THE TCP THREE-WAY HANDSHAKE
--------------------------------------------------------------------------------
PURPOSE: Identify the SYN → SYN-ACK → ACK sequence that establishes every TCP
connection. Understand how to distinguish a successful connection from a refused
or dropped one.

BACKGROUND
  Every TCP connection begins with a three-packet handshake before any data is
  transferred. If you see a SYN with no following SYN-ACK, the server is
  unreachable or the connection was blocked. If you see a RST (reset) packet,
  the connection was forcibly terminated. These two patterns are the primary
  diagnostic signals when troubleshooting connectivity failures.

STEPS
  1.  Start a new capture in Wireshark on your active interface
  2.  Open a browser and navigate to: http://neverssl.com
      This site is purpose-built to never redirect to HTTPS, making it ideal
      for observing plaintext TCP traffic and the handshake in Wireshark.
  3.  After the page loads, stop the capture
  4.  Open a terminal and run:
         nslookup neverssl.com
      Note the IP address returned — it changes, so always look it up fresh.
  5.  In the Wireshark filter bar, type:
         tcp and ip.addr == <IP from nslookup>
      Press Enter
  6.  Find the three-packet sequence in order:

         Packet 1  Flags: SYN          Your machine: "I want to connect. Here
                                        is my sequence number."
         Packet 2  Flags: SYN, ACK     Server: "I got your request. Here is my
                                        sequence number. Connection accepted."
         Packet 3  Flags: ACK          Your machine: "Got it. Connection open.
                                        Ready to send data."

  7.  Click each packet in the three-packet sequence and confirm the Flags field
      in the Packet Detail pane matches the above

WHAT TO RECORD FOR YOUR PORTFOLIO
  - Screenshot showing the three-packet SYN → SYN-ACK → ACK sequence in the
    packet list with the Flags column visible

SECTION 9: EXERCISE C — SPOT CLEARTEXT CREDENTIALS (HTTP)
--------------------------------------------------------------------------------
PURPOSE: Demonstrate that credentials submitted over HTTP (without TLS) are
readable in plaintext by anyone who can see the network traffic.

EDUCATIONAL NOTICE: This exercise must only be performed on systems you own or
have explicit written permission to test. Never use packet capture against
systems or networks you do not control.

BACKGROUND
  HTTP transmits all data as plaintext. This includes the contents of login
  forms — usernames and passwords are sent as readable text inside the POST
  request body. HTTPS adds TLS encryption, which encrypts the payload so it
  cannot be read even if captured. This exercise demonstrates the exact
  vulnerability that made HTTPS mandatory for any site handling sensitive data.

SETUP
  No additional software required. This exercise uses a publicly accessible
  deliberately vulnerable test banking application maintained for security
  education. It runs over plain HTTP with no TLS.

  Target URL: http://zero.webappsecurity.com/login.html

NOTE ON AZURE BACKGROUND TRAFFIC
  Azure VMs generate continuous HTTP POST traffic to 168.63.129.16 (Azure's
  internal health/metadata service). This will appear in your capture and
  can obscure the login POST. The filter in Step 5 accounts for this.

STEPS
  1.  Open Wireshark and start a capture on your active Ethernet interface
      (the same interface used in Exercises A and B — not loopback)
  2.  Open a browser and navigate to:
        http://zero.webappsecurity.com/login.html
  3.  Enter any test credentials and click Sign In
        (e.g. Username: testlogin | Password: Fakepassword1)
  4.  Stop the capture
  5.  In the filter bar, type:
        http.request.method == POST and http.host contains "zero"
      Press Enter — this isolates the login POST and excludes Azure
      background traffic to 168.63.129.16
  6.  Click the POST packet
  7.  In the Packet Detail pane, expand:
        "Hypertext Transfer Protocol" → "HTML Form URL Encoded"
  8.  You will see the username and password fields in plaintext inside the packet

WHAT TO RECORD FOR YOUR PORTFOLIO
  - Screenshot of the POST packet with the HTML Form URL Encoded section expanded
    showing the plaintext credentials
  - This screenshot is compelling portfolio evidence of why HTTPS is mandatory

SECTION 10: EXERCISE D — FOLLOW A FULL TCP STREAM
--------------------------------------------------------------------------------
PURPOSE: Reassemble a complete HTTP conversation from individual packets into
a human-readable client/server dialogue.

BACKGROUND
  Individual packets are fragments. A TCP stream is the complete conversation
  reassembled in order. Incident responders use stream reconstruction to
  understand what data was sent, what commands were issued, and how the server
  responded — all from a raw packet capture.

STEPS
  1.  Start a capture on your active interface
  2.  Open a browser and navigate to any HTTP website (e.g. http://neverssl.com)
  3.  Stop the capture after the page loads
  4.  In the filter bar, type:
         http
      Press Enter to isolate HTTP traffic
  5.  Right-click any HTTP packet in the list
  6.  Select: Follow → TCP Stream
  7.  Wireshark opens a Stream Content window showing the full conversation:
         Red text  = your browser's outbound request (HTTP GET headers, etc.)
         Blue text = the server's inbound response (HTTP headers + HTML body)
  8.  Read through the stream — identify the GET request, the response headers,
      and the HTML content in the body

WHAT TO RECORD FOR YOUR PORTFOLIO
  - Screenshot of the TCP Stream window with both the request (red) and response
    (blue) visible

SECTION 11: SAVE AND EXPORT CAPTURES
--------------------------------------------------------------------------------
Save your captures as .pcapng files. These are your portfolio artefacts and
can be reopened in Wireshark at any time for further analysis.

SAVE THE FULL CAPTURE
  File → Save As → choose .pcapng format
  Name your files descriptively:
    dns-lookup-google.pcapng
    tcp-handshake-example.pcapng
    http-cleartext-creds.pcapng
    tcp-stream-follow.pcapng

EXPORT ONLY FILTERED PACKETS
  Apply a display filter to narrow the view
  File → Export Specified Packets → select "Displayed" → Save

RE-OPEN A SAVED CAPTURE
  File → Open → select the .pcapng file
  All packets will be restored exactly as captured

COMMAND-LINE CAPTURE WITH TSHARK
  tshark is the terminal-based version of Wireshark, included in the
  installation. Use it to capture on remote servers or automate captures.

  See scripts\Invoke-TsharkCapture.ps1 for a PowerShell wrapper that:
    - Detects the active network interface
    - Runs a timed tshark capture
    - Saves the output to a named .pcapng file

  Manual tshark usage:
    tshark -i eth0 -w capture.pcapng -c 1000
      -i   network interface name
      -w   output file path
      -c   stop after this many packets
